# Further Testing Plan

## Goal

Close the remaining unknowns that cannot be resolved from source review and partial Supabase metadata alone.

## Highest-Value Follow-Up Checks

### 1. Storage review

Objective:

- confirm whether any storage buckets are public
- confirm whether object reads or writes are exposed to `anon` or `authenticated`

Actions:

- inspect `storage.buckets`
- inspect storage policies on `storage.objects`
- verify whether public URLs are enabled anywhere operationally

Expected output:

- bucket names
- public/private status
- list of read/write policies

### 2. Narrow grant review for public schema

Objective:

- verify direct grants on only the application tables

Actions:

- re-run grants query filtered to `public`
- focus on `blips`, `bytes`, `blip_views`, `byte_views`, `bloq_views`, `project_views`, `claps`, `visits`

Expected output:

- exact grants by role for those tables only

### 3. Validate function execution surface

Objective:

- determine which roles can execute privileged functions

Actions:

- query routine grants for public functions
- verify whether `anon` or `authenticated` can execute clap/view/visit helper functions directly

Expected output:

- function grant matrix

### 4. Confirm REST exposure

Objective:

- determine which tables are reachable through Supabase autogenerated APIs in practice

Actions:

- test read/write against likely exposed tables using anon credentials in a safe non-production environment
- specifically validate `project_views`, `claps`, and `visits`

Expected output:

- list of direct API operations that succeed or fail

### 5. Validate actual production headers

Objective:

- confirm whether infrastructure injects security headers that are absent from repo config

Actions:

- inspect deployed responses for main pages and API routes
- record CSP, frame protections, referrer policy, HSTS, content-type protections

Expected output:

- response header matrix for main endpoints

### 6. Dependency review

Objective:

- identify known vulnerabilities or high-risk package exposure

Actions:

- run dependency audit on lockfile
- review packages that process content, markdown, remote network calls, or bot/webhook traffic

Expected output:

- dependency findings with severity and upgrade path

## Recommended Order

1. storage review
2. public-schema grants and function grants
3. direct REST exposure testing
4. production header inspection
5. dependency audit

## Completion Criteria

This phase is complete when:

- all public data surfaces are explicitly classified
- all privileged functions have known callers and grants
- all storage buckets are classified
- production browser/API headers are known
- remaining risk is limited to accepted tradeoffs rather than unknowns
