Root Cause: The Migration Plan's Fatal Assumption

From .kilo/plans/1775990254226-gentle-sailor.md (line 65):
"Written by triggers only — bypass RLS ✅"
This is wrong. Postgres trigger functions execute at the privilege level of their creator, not the table owner. The default SECURITY INVOKER mode (which was the state of all 11 trigger functions) means they run with the privileges of the session user — the end user's JWT. RLS applies to that user. Only SECURITY DEFINER functions inherit the owner's privileges and bypass RLS.

Why testing didn't catch this: The plan was tested in the Supabase SQL editor, which runs as postgres (superuser). As superuser, RLS is always bypassed regardless of SECURITY INVOKER/DEFINER. The bug only manifests when queries execute under a real user's JWT — exactly what the Next.js API routes do via getServerSupabase(user.accessToken).

Phase 4: Deep Dive — The PostgreSQL 17.6 auth.uid() Inlining Bug

Diagnostic Investigation

To determine why auth.uid() = user_id fails in policy expressions, a diagnostic function was created:

CREATE OR REPLACE FUNCTION public._rls_debug_fn(input_user_id uuid)
RETURNS boolean
LANGUAGE plpgsql
VOLATILE
SECURITY INVOKER
AS $$
DECLARE
  uid uuid;
BEGIN
  uid := auth.uid();
  RAISE EXCEPTION '[RLS DEBUG] auth.uid()=% · input_user_id=% · match=%',
    uid, input_user_id, (uid = input_user_id);
END;
$$;
Critical discovery: When auth.uid() is called through a PL/pgSQL wrapper, it resolves correctly. When called inline in a LANGUAGE sql STABLE policy expression, it fails.

Why: auth.uid() is LANGUAGE sql STABLE. PostgreSQL's query planner can inline STABLE SQL functions into the query. When inlined, the coalesce fallback from NULL::request.jwt.claim.sub to request.jwt.claims::jsonb->>'sub' may not fire correctly in the policy evaluation context. PL/pgSQL functions cannot be inlined by PostgreSQL — they force proper function call execution through the executor, where the coalesce fallback works.

Plan: PL/pgSQL Wrapper (fix-rls-counter-trigger-pg-176.md)

Replace inline auth.uid() = user_id in all RLS policies with a PL/pgSQL wrapper function:

CREATE OR REPLACE FUNCTION public.rls_user_matches_auth_uid(input_user_id uuid)
RETURNS boolean
LANGUAGE plpgsql
VOLATILE
SECURITY INVOKER
SET search_path = public, auth
AS $$
BEGIN
  RETURN auth.uid() = input_user_id;
END;
$$;
DECISION
Adopt PL/pgSQL wrapper (rls_user_matches_auth_uid) for all RLS policy expressions on user_interaction_counters. This forces proper auth.uid() evaluation through the PL/pgSQL executor, preventing inlining of the LANGUAGE sql STABLE auth.uid() function.

The Re-Like Failure: Missing SELECT Policy

Problem: After deploying the PL/pgSQL wrapper policies, first like works, unlike works, but re-like still fails.

Root cause: INSERT ... ON CONFLICT DO NOTHING needs to read existing rows to detect the conflict. Under RLS, reading requires a SELECT policy. The counter table had INSERT and UPDATE policies — but no SELECT policy.

PostgreSQL 17.6 behavior: WITH CHECK is evaluated BEFORE conflict detection. Even if the conflict check would find the row, the WITH CHECK fails first (permission denied for the SELECT required to check conflicts).

Phase 5: The Missing Piece — SELECT Policy

Migration: add_counter_select_policy

ON public.user_interaction_counters
FOR SELECT
TO public
USING (public.rls_user_matches_auth_uid(user_id));

Key Discoveries (Technical Deep Dive)

Discovery #1: auth.uid() Inlining in Policy Expressions

PostgreSQL 17.6 behavior: When auth.uid() (declared LANGUAGE sql STABLE) is used directly in RLS policy expressions, PostgreSQL's query planner may inline the function body into the query. This optimization skips the coalesce fallback from NULL::request.jwt.claim.sub to request.jwt.claims::jsonb->>'sub' — the function resolves to NULL instead of the actual user ID.

Why PL/pgSQL works: PostgreSQL cannot inline LANGUAGE plpgsql functions. They must be called through the full executor, which properly evaluates the coalesce fallback chain.

Discovery #2: SELECT Policy Required for ON CONFLICT

In PostgreSQL: INSERT ... ON CONFLICT DO NOTHING requires SELECT permission on the target table to detect whether a conflicting row exists. Under RLS, this means a SELECT policy is required — not just an INSERT policy.

Why the first like succeeds: No existing counter row → no conflict to detect → no SELECT needed → INSERT policy alone suffices.

Why re-like fails: Counter row exists → must SELECT to check conflict → no SELECT policy → 42501.

Discovery #3: request.jwt.claim.sub Is NULL in Triggers

Inside trigger functions, request.jwt.claim.sub is NULL. Only request.jwt.claims (the full JSON) is available. This is why the coalesce chain in auth.uid() matters — the function falls back from request.jwt.claim.sub (NULL in triggers) to request.jwt.claims->>'sub' (works in triggers) before falling back to NULLIF(current_setting('request.jwt.claim.sub', true), '')::uuid.

Discovery #4: WITH CHECK Evaluated Before Conflict Detection

PostgreSQL 17.6 execution order for INSERT ... ON CONFLICT DO NOTHING:
1. Evaluate WITH CHECK policy
2. Check for conflict
3. If conflict → DO NOTHING; if no conflict → INSERT

This means the WITH CHECK must pass even when the row will ultimately be a no-op due to conflict. The SELECT required for step 2 is a separate permission from the INSERT WITH CHECK.

Clarification #3: What Is an RLS Policy?

An RLS policy is a boolean expression attached to a table — not a function.

Policy: A rule that says "for this operation, allow rows where this expression is true."
Function: A callable block of code. The policy expression can reference functions, but the policy itself is metadata on the table, not a function. 
-- THIS is a policy (an expression/rule on the table):
CREATE POLICY "Users can insert own counter"
ON user_interaction_counters FOR INSERT
WITH CHECK (rls_user_matches_auth_uid(user_id));

-- THIS is a function (called by the policy expression):
CREATE FUNCTION rls_user_matches_auth_uid(input_user_id uuid)
RETURNS boolean
LANGUAGE plpgsql
AS $$
BEGIN
  RETURN auth.uid() = input_user_id;
END;
$$;
In the Supabase dashboard: when you go to a table → "Policies," you're seeing these boolean expressions. When a query hits the table, PostgreSQL evaluates the matching policy expression for each row. If the expression returns true, the row is accessible. If false, the row is silently excluded (UPDATE/DELETE/SELECT) or the row is rejected (INSERT).

Clarification #4: Why the SELECT Policy Exists (It's Not for User Reads)

The SELECT policy is not for users reading their counter data from the frontend. Counter reads go through SECURITY DEFINER RPCs (get_top_user, get_interaction_analytics) that bypass RLS entirely.

The SELECT policy exists solely to enable the conflict detection step inside INSERT ... ON CONFLICT DO NOTHING. Without a SELECT policy, PostgreSQL refuses to check whether a conflicting row already exists → 42501.

Clarification #5: WITH CHECK Before Conflict Detection — Why It Matters

This execution order is functionally critical — not "aesthetic" or "just logical flow."

PostgreSQL 17.6 evaluates INSERT ... ON CONFLICT DO NOTHING in this fixed order:
1. WITH CHECK policy (INSERT policy)
2. Conflict detection (requires SELECT permission)
3. If conflict → DO NOTHING; else → INSERT

The implication: Even when a row already exists (so the INSERT will ultimately be a no-op), the WITH CHECK must pass first. If auth.uid() returns NULL (inlining bug in Approach #2), the WITH CHECK fails at step 1 — you never reach step 2. The operation errors out even though nothing actually needs to be inserted.

Clarification #6: JWT Flow — Next.js to PostgreSQL Session

How the user's JWT reaches the database and enables RLS: 
Browser
  → HTTP request with cookie (auth-token)
    → Next.js API route (src/app/api/likes/route.ts)
      → AuthManager.getAuthenticatedUser()
        ← reads cookie, decodes JWT payload, checks exp expiry
        ← extracts user.id + user.accessToken
        → getServerSupabase(user.accessToken)
          → createClient(url, anonKey, { headers: { Authorization: "Bearer <jwt>" } })
            → Supabase client sends query to PostgREST (Supabase's REST API layer)
              → PostgREST validates JWT signature, extracts claims
                → PostgREST sets request.jwt.claims + request.jwt.claim.sub in the PG session

Key insight: The auth check is not between Next.js and the Supabase client. The Supabase client simply passes the JWT as a header. The real validation — JWT signature verification + claim extraction — happens at PostgREST. PostgREST injects the claims into the PostgreSQL session, and then RLS evaluates them. RLS is the gatekeeper at the database level.

Clarification #7: The auth.uid() Coalesce Chain — Two Fallback Paths

auth.uid() tries two different session settings to find the user's UUID. Both are needed because one of them is NULL in triggers. 
CREATE OR REPLACE FUNCTION auth.uid()
RETURNS uuid
LANGUAGE sql
STABLE
AS $$
SELECT coalesce(
    nullif(current_setting('request.jwt.claim.sub', true), ''),          -- Path A
    (nullif(current_setting('request.jwt.claims', true),                 -- Path B
     '')::jsonb ->> 'sub')
)::uuid
$$;
SettingInside Trigger Functions?When Is It Set?Notes
request.jwt.claim.sub (Path A) NULL Set by PostgREST for direct queries only This is why coalesce is needed — Path A fails in triggers
request.jwt.claims (Path B) Available ✅ Set by PostgREST for all queries (the full JSON blob) Always available. Path B is what actually works in triggers.
What went wrong: When auth.uid() is inlined by PostgreSQL's query planner (Approach #2), the function body is merged into the calling query. The coalesce doesn't properly evaluate both branches in the policy evaluation context → the function returns NULL instead of the UUID. When called through a PL/pgSQL wrapper (Approach #3), the executor calls it properly → coalesce works → Path B resolves the sub claim → returns correct UUID.

Clarification #9: Silent Bug Window — All Three Factors

The bug went undetected for ~2 months. All three of these factors contributed:
Factor 1 — Existing counter rows survived the migration. Users who already had a counter row before April 12 never hit the INSERT path for their existing interactions. The UPDATE path (for decrementing counters) silently affects 0 rows — no error thrown.
Factor 2 — UPDATE with zero RLS policies = silent success. Unlike and other operations that UPDATE the counter table don't throw 42501. They match 0 rows and silently succeed. This made it appear that the system was working normally for 2 months.
Factor 3 — Client error swallowing. The API handler caught the 500 Internal Server Error and returned a generic message. The client-side like-service.ts displayed "Network error occurred." Users blamed their internet connection, not a backend bug. The actual 42501 error code was hidden in server logs that no one was monitoring.

Clarification #10: Trigger Architecture — Why the Counter Table Gets Updated

Every time a user likes, comments, or interacts with father_son_stories, a database trigger atomically updates their counter row. This guarantees consistency — you cannot create a like without the counter updating. 
User clicks ❤️ (like):
  → INSERT INTO likes (page_path, user_id) VALUES (...)
    → RLS on likes approves ✅
    → 🔥 TRIGGER fires: handle_like_insert()
      → ensure_user_counter_row(NEW.user_id)
        → INSERT INTO user_interaction_counters ON CONFLICT DO NOTHING
        → RLS must approve: SELECT (conflict check) + INSERT (new row)
      → UPDATE user_interaction_counters
        SET like_count = like_count + 1, updated_at = now()
        WHERE user_id = NEW.user_id
        → RLS must approve: UPDATE (USING + WITH CHECK)

User clicks 💔 (unlike):
  → DELETE FROM likes WHERE id = ?
    → RLS on likes approves ✅
    → 🔥 TRIGGER fires: handle_like_delete()
      → UPDATE user_interaction_counters
        SET like_count = GREATEST(like_count - 1, 0), updated_at = now()
        WHERE user_id = OLD.user_id
        → RLS must approve: UPDATE (USING + WITH CHECK)
Architectural pattern: Using database triggers for counter updates guarantees atomicity — the counter update is part of the same transaction as the source operation. No app code can forget to call incrementCounter(). The database enforces the invariant at all times.

This same pattern applies to comments (incrementing comment_count) and father_son_stories (incrementing story_count). All three source tables have triggers that update the same counter table.
