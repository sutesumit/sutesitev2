## The Thing I Built

> "The best interface is the one you already have open."

So I have these [blips](/blip). Short posts, 280 characters. Like tweets, but mine. No algorithm. No engagement metrics. No "you might also like." Just thoughts, in order, on a page I own.

Posting one used to mean curl. Terminal. API key. Fine for a robot, annoying for a human on a phone walking to get coffee.

Now there are two interfaces. Terminal commands for when I'm working and don't need Telegram distractions. And Telegram bot for when I'm already distracted. Same database, same blips, different entry points depending on where I am.

This article is about the Telegram bot. How it works, what I learned building it. The curl stuff is boring. HTTP calls. You know HTTP.

---

## Telegram Bots, Quickly

You talk to [@BotFather](https://t.me/botfather) on Telegram. Yes, that's a real account. Yes, the name is ridiculous. Send `/newbot`, pick a name, get a token.

That token is your bot's password. Lose it and the bot is orphaned forever. Telegram doesn't do password resets. Parental rights are permanent.

With the token, you call Telegram's API. `sendMessage` to reply. `getUpdates` to fetch messages. It's HTTP. JSON in, JSON out. Nothing fancy, just REST API.

Bots are peculiar accounts. They won't message you first, they speak only so fast (rate limits), and they mostly listen for commands (the `/` kind). Otherwise, they're just code talking to Telegram's servers and masquerading as a user.

## Polling vs Webhooks

Okay so someone messages your bot. How does your code know?

Two ways.

### Polling

Your code keeps asking: "Any new messages?"

```text
You: Anything?
Telegram: No.
(wait)
You: Anything?
Telegram: No.
(wait)
You: Anything?
Telegram: No.
(wait)
You: Anything?
Telegram: YES FINALLY User 123 said "hello".
```

It works. It's simple. The downside is you're making calls constantly, even when nobody loves your bot enough to message it. And there's a small delay between when the message arrives and when you ask for it.

Think of it like checking the fridge. You keep opening it, hoping something new appeared. Nothing does. But you check anyway.

### Webhooks

You give Telegram a URL. When a message arrives, Telegram POSTs to that URL.

```text
User sends message → Telegram → POST to your URL → You handle it
```

No polling. No wasted calls. Real-time. Your server sleeps until there's work.

The catch: your URL needs to be public. Telegram can't POST to `localhost:3000`.

### What Webhooks Actually Are

A webhook is just: someone else's server calling your server.

That's it. It's an HTTP POST request. You expose an endpoint. They hit it when something happens.

You've seen this pattern before:

- **Stripe:** Payment succeeds → POST to your `/webhooks/stripe` endpoint
- **GitHub:** Push to repo → POST to your `/webhooks/github` endpoint
- **Slack:** Someone mentions @bot → POST to your `/webhooks/slack` endpoint
- **Telegram:** Someone messages bot → POST to your `/webhooks/telegram` endpoint

Same pattern. Different services. They all push events to you instead of you pulling from them.

### Rule of Thumb

**Use webhooks when:** You control a server and want real-time updates from external services.

**Use polling when:** You're behind a firewall, don't have a public URL, or the service doesn't support webhooks.

Most production apps use webhooks. Most local development uses polling (or a tunnel like ngrok to fake a public URL).

### What I Use

Webhooks in production. The bot lives at `/api/telegram/webhook`. Telegram POSTs there.

Polling exists for local dev. Run `npm run bot`, it polls. Useful before you have HTTPS set up. I don't use it anymore.

## Grammy and Middleware

I use [grammy](https://grammy.dev/) for the bot. TypeScript framework. Handles the API plumbing.

You know middleware. Express has it. Redux has it. It's everywhere. A chain of functions, each doing one thing, passing control to the next.

### When Middleware Helps

Cross-cutting concerns. Stuff that applies to multiple handlers:

- Auth (is this user allowed?)
- Logging (what happened?)
- Error handling (what broke?)
- Rate limiting (slow down)

Without middleware, each handler repeats this logic. With middleware, it's centralized. Register once, applies everywhere.

### When Middleware Hurts

Over-abstraction. Not everything needs a chain.

If you have two handlers and they share nothing, middleware is overhead. You're adding complexity for theoretical future flexibility that never arrives.

I've built middleware systems for apps with three routes. Waste of time. The indirection cost more than the duplication would have.

### This Bot

The bot has:

- 7 command handlers (`/start`, `/byte`, `/blip`, `/list`, `/get`, `/edit`, `/del`)
- 1 message handler (catch-all for creating bytes)

Shared logic: auth check. That's it. Every handler checks if the user ID is whitelisted.

Could I use middleware for this? Yes. Grammy supports it:

```text
bot.use(authMiddleware)
bot.command('start', handler)
bot.on('message', handler)
```

Did I? No. I put the check in each handler. Seven lines, repeated eight times.

Why? Because the bot is small. The auth check is one line. If it grows to ten handlers, I'll extract it. Until then, explicit is fine. The duplication is visible. The abstraction would hide it.

### The Real Lesson

Middleware isn't good or bad. It's a tradeoff.

**Pros:** DRY, centralized logic, easy to add new cross-cutting concerns.
**Cons:** Indirection, harder to trace execution, overkill for small apps.

My instinct now: wait for the third duplication. First time: just write it. Second time: copy it. Third time: maybe extract. Often the third time never comes.

### The Order Bug

Grammy runs handlers in registration order. First match wins.

I wrote this:

```typescript
// Wrong order
bot.on('message', async (ctx) => {
  // This catches EVERY message, including /start, /list, etc.
  const text = ctx.message.text
  await createByte(text)
})

bot.command('start', async (ctx) => {
  // Never reached. The message handler already matched.
  await ctx.reply('Hello!')
})
```

Sent `/start` to the bot. It created a byte with content "/start".

Not what I wanted.

The `bot.on('message')` handler matches any message. Commands are messages. So `/start` hit the message handler first, got treated as byte content, done. The command handler never saw it.

Fixed:

```typescript
// Right order
bot.command('start', async (ctx) => {
  await ctx.reply('Hello!')
})

bot.command('list', async (ctx) => {
  const bytes = await getBytes()
  await ctx.reply(formatBytes(bytes))
})

// ... other commands ...

bot.on('message', async (ctx) => {
  // Now this only catches non-command messages
  const text = ctx.message.text
  await createByte(text)
})
```

Specific first. Generic last. The catch-all only runs if nothing else matched.

This isn't specific to grammy. Express routes work the same way. So do Redux reducers. So does any system where you register handlers that match patterns. Order matters when multiple patterns could match the same input.

## No Sessions

The bot doesn't use sessions. At all.

Originally I thought `/edit` would be conversational:

```text
Me: /edit a3
Bot: Current content: "old thought". Send new content:
Me: new thought
Bot: Updated!
```

Two messages. State between them. The bot remembers we're in "edit mode" for byte a3.

Then I thought: why?

```text
Me: /edit byte a3 new thought
Bot: Updated!
```

One message. Parse the type, serial, and new content. Done. No state. No memory. No "cancel" command for abandoned edits. No timeout logic. No session cleanup.

The bot is stateless. Each message is independent. It doesn't remember the previous message. It doesn't care about the next one. It just handles what's in front of it.

This is simpler. Whether it's better depends on what you're building. For a CRUD bot posting 280-character thoughts, stateless is the way.

## Auth

The bot is public. If you know the bot's username, you can find it on Telegram. Anyone can send it a message.

But only whitelisted users get a response.

Telegram sends the user's ID with every message. Can't be faked. I check it against a list:

```text
TELEGRAM_ALLOWED_USER_IDS=123456789,987654321
```

If your ID matches, you're in. If not, "Unauthorized."

No JWT. No OAuth. No login flow. Just: is this number in the list?

For a personal bot, this is enough. The bot is public, the API is private.

### Wait, Aren't Webhooks Public Too?

Yes. The webhook URL (`/api/telegram/webhook`) is public. Anyone could POST to it.

But here's the thing: Telegram signs every webhook request with a secret. You verify the signature before processing.

```text
POST /api/telegram/webhook
Header: X-Telegram-Bot-Api-Secret-Token: <secret>
Body: { "message": { ... } }
```

You set the secret when you register the webhook. Telegram includes it in every request. You check it on your end. If it doesn't match, you reject the request.

This is how production webhooks work:

- **Stripe:** Signs requests with your webhook secret. You verify the signature.
- **GitHub:** Sends a hash of the payload with your secret. You verify the hash.
- **Slack:** Includes a timestamp and signature. You verify both.

The pattern: public URL, private verification. Anyone can knock. Only the right key opens the door.

For this bot, I skipped signature verification. Lazy? Maybe. But the user ID whitelist already gates access. Even if someone forged a Telegram request, they'd need a whitelisted user ID to do anything useful.

## The Commands

```text
| Command                    | What it does              |
|----------------------------|----------------------------|
| `/start`                   | Help text                  |
| `/byte <content>`          | Create a short thought     |
| `/blip <term>:<meaning>`   | Create a term:meaning pair |
| `/list <byte|blip>`        | List recent items          |
| `/get <byte|blip> <serial>`| Show specific item         |
| `/edit <byte|blip> <serial> <content>` | Update item    |
| `/del <byte|blip> <serial>`| Delete item               |
| any text                   | Create new byte            |
```

All single-message. No conversations. Stateless.

There are two content types:

- **Bytes**: Short thoughts, just text. Created with `/byte` or by sending any message.
- **Blips**: Term:meaning pairs, like dictionary entries. Created with `/blip term:meaning`.

## The Menu

Telegram bots can register a command menu. Type `/` and suggestions appear.

```typescript
await bot.api.setMyCommands([
  { command: "start", description: "Show help" },
  { command: "byte", description: "Create a byte (short thought)" },
  { command: "blip", description: "Create a blip (term:meaning)" },
  { command: "list", description: "List bytes or blips" },
  { command: "get", description: "Get a byte or blip" },
  { command: "edit", description: "Edit a byte or blip" },
  { command: "del", description: "Delete a byte or blip" },
])
```

{/*
SCREENSHOT PLACEHOLDER: Telegram command menu showing all available commands when typing /
*/}

One catch: the bot instance is cached across webhook calls. `setMyCommands` only ran on first init. After deploying, the menu didn't appear.

Fixed by calling it on every webhook, cached or not:

```typescript
if (botInstance) {
  await botInstance.api.setMyCommands([...])
  return botInstance
}
```

It's idempotent. No harm in calling it repeatedly.

## Channel Broadcast

Every new byte and blip gets posted to [@blipbotlive](https://t.me/blipbotlive). A public channel. Anyone can follow.

The broadcast happens right after the item is created:

```typescript
const byte = await createByte(text)
await ctx.reply(replies.byteCreated(byte.byte_serial))

const channelId = process.env.TELEGRAM_CHANNEL_ID
if (channelId) {
  await bot.api.sendMessage(
    channelId,
    replies.channelBlip(byte.byte_serial, byte.content),
    { parse_mode: "HTML" }
  )
}
```

Multiple entry points, one broadcast:

1. **Telegram bot `/byte` command.** Creates byte, then broadcasts.
2. **Telegram bot direct message.** Creates byte, then broadcasts.
3. **Telegram bot `/blip` command.** Creates blip, then broadcasts.
4. **CLI via API route.** Creates blip, then broadcasts.

Same channel. Same format. Doesn't matter how the item was created.

The bot needs to be a channel admin with "Post Messages" permission. I learned this the hard way. Items weren't showing up. Checked logs. Nothing. Turns out Telegram silently drops messages from non-admin bots. No error. Just silence.

## What I Learned

**Webhooks over polling for production.** Polling works, but webhooks are cleaner. Your server only wakes up when there's work.

**Middleware order matters.** Register specific handlers before generic ones. The chain runs in order. Learn this by breaking it once.

**Stateless is simpler.** Sessions add complexity. If you can avoid them, avoid them. `/edit byte a3 text` is one message. No state needed.

**User ID is free auth.** Telegram sends it. Use it. Keep your JWTs for something else.

**Bot admins need permissions.** A bot can't post to a channel unless it's an admin with post permissions. Telegram doesn't error. It just drops the message. Check permissions first.

## Two Interfaces

There are two ways to post now.

**Terminal.** Curl commands hitting the API directly. `blip "content=thought"` creates one. `blips` lists them. Fast when I'm already in a terminal, which is most of the time when working.

**Telegram.** The bot. Open the app, type, done. Fast when I'm not at a computer. Walking, in bed, at a cafe.

Both hit the same database. Both broadcast to the channel. Both are instant. No switching contexts, no waiting, no friction.

The curl commands live in a separate file. Shell aliases. `blip`, `blips`, `blip-get`, `blip-edit`, `blip-del`. I'm not putting them in this article because they're not interesting. Just HTTP calls with an auth header. That's it. That's the whole thing. Okay fine, the real reason is security.

## Why This Matters

The whole point: I can post a thought the moment I form it.

Before: idea strikes → try to remember it → find terminal → remember the curl syntax → craft the command → post. By step 3, the idea might be gone. Or diluted. Or I got distracted by something else. Or I convinced myself it wasn't that good anyway.

Now: idea strikes → open Telegram → type → done. Three seconds. The thought is captured before my brain can talk me out of it.

This is what good tools do. They get out of the way. They reduce the gap between "I have a thought" and "the thought is saved."

The bot isn't the point. The bytes aren't the point. The blips aren't the point. The architecture isn't the point. Capturing thoughts while they're fresh is the point. Everything else is just infrastructure.

---

## A Note on Security

Both interfaces require auth. The API uses a secret key in a header. The bot uses a whitelist of Telegram user IDs.

The security model is simple: if you have my phone or my laptop, you can post. If you have neither, you can't.

That's it. That's the whole security model. No zero-trust, no defense in depth, no principle of least privilege. Just: physical access to my devices.

Is this enterprise-grade? No. Is it sufficient for a personal microblogging tool? Absolutely.
