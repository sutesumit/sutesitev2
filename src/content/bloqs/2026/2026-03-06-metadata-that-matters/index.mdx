## The Orphan Problem, Again

> "A page without metadata is a stranger at its own party."

I noticed something embarrassing. Every bloq post on my site — when shared on Twitter, LinkedIn, or indexed by search engines — showed the same thing:

```
Title: sumit sute
Description: Sumit Sute's personal dev page...
```

Not the article title. Not the article summary. Just the generic homepage metadata, copy-pasted across every URL like a lazy form letter.

The content was unique. The code was thoughtful. But to the outside world, every post looked identical. *The Skill That Writes Itself*? Same metadata as *Hello World*. *Architecture Was Already There*? Indistinguishable from my homepage.

I'd built the reading experience. I'd ignored the discovery experience.

---

## The Vocabulary of Platforms

The web has a translation problem. Your page speaks HTML. But Twitter wants one dialect. LinkedIn wants another. Google wants structured data. Pinterest wants something else entirely.

These aren't just different formats — they're different *ontologies*. Different ways of understanding what a thing *is*.

### OpenGraph: The Social Dialect

OpenGraph was invented by Facebook in 2010. The idea: any web page can describe itself in a way social platforms understand. Instead of guessing what your page is about, platforms read explicit metadata.

```html
<meta property="og:title" content="The Skill That Writes Itself" />
<meta property="og:description" content="An orphan article..." />
<meta property="og:type" content="article" />
<meta property="og:image" content="https://..." />
```

When someone pastes your URL into Twitter or LinkedIn, the platform fetches these tags and builds a preview card. The image, the title, the description — all come from OpenGraph.

**Why images matter:** The `og:image` is your first impression on social. A good preview image can double click-through rates. But here's the thing — if you don't provide one, platforms will either show nothing (boring) or scrape something random from your page (often worse).

I decided not to fake it. If a post has no image, it gets no image. Better silence than a misleading homepage screenshot.

### JSON-LD: The Search Dialect

JSON-LD stands for "JSON for Linking Data." It's JSON, but with a twist: it's *linked* data, meaning it connects your content to a shared vocabulary.

Regular JSON is just structure:

```json
{
  "title": "My Post",
  "author": "Sumit Sute"
}
```

JSON-LD adds *context*:

```json
{
  "@context": "https://schema.org",
  "@type": "BlogPosting",
  "headline": "My Post",
  "author": {
    "@type": "Person",
    "name": "Sumit Sute"
  }
}
```

The `@context` says: "I'm using the schema.org vocabulary." The `@type` says: "This is a BlogPosting, not a Product or Event." Search engines don't have to guess — they *know*.

This is how Google builds those rich results. The knowledge panel that shows cooking time for recipes. The event dates for concerts. The review stars for products. All of it comes from structured data.

**The key difference from regular JSON:** JSON-LD isn't about storing data. It's about *asserting meaning*. You're not just saying "there's an author" — you're saying "this is a Person, in the schema.org sense, and here's their name."

## The Canonical Problem

There's another piece: canonical URLs.

When someone links to your post with query parameters (`?utm_source=twitter`), or when a scraper copies your content, search engines see multiple URLs with identical content. Which one is the "real" one?

```html
<link rel="canonical" href="https://sumitsute.com/bloq/the-skill-that-writes-itself" />
```

This tells search engines: "No matter how you got here, this is the authoritative URL." It consolidates ranking signals. It protects against duplicate content penalties. It's a small thing that matters at scale.

## The Mental Model

Here's how I think about it now:

| Layer | Speaks to | Purpose |
|-------|-----------|---------|
| HTML | Browsers | Renders the page |
| OpenGraph | Social platforms | Builds preview cards |
| JSON-LD | Search engines | Enables rich results |
| Canonical | Everyone | Declares the authoritative URL |

Each layer is a translation. Each layer is optional but valuable. Each layer requires you to think about how your content exists *outside* your site.

---

## What I Built

The implementation was straightforward once I understood the concepts:

1. **Per-page metadata** — Each bloq post generates its own title, description, and canonical URL
2. **Conditional images** — If a post has an image, include it in OpenGraph. If not, don't fake it with a fallback
3. **JSON-LD BlogPosting** — Each post asserts its type, author, dates, and keywords in schema.org vocabulary
4. **Static generation** — Build once, serve fast. Search engines get immediate content, no JavaScript required

The code is unremarkable. The *understanding* was the work.

---

## What I Took Away

- **Metadata is translation.** You're not just adding tags — you're teaching platforms how to understand your content.
- **OpenGraph is your social identity.** When someone shares your link, the preview *is* your content. Make it count.
- **JSON-LD is assertion, not storage.** The `@context` and `@type` matter more than the data itself. You're participating in a shared vocabulary.
- **Don't fake images.** A missing image is honest. A misleading fallback is not.
- **Canonical URLs are hygiene.** They take five minutes and protect you from a class of SEO problems you don't want to debug.

The bloq now speaks for itself. Each article has its own voice, its own preview, its own identity in the machines that distribute it.

The metadata writes itself. I just had to learn the language.
