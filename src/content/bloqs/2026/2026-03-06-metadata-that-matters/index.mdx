## The Problem

I shared one of my bloq posts on LinkedIn and noticed the preview card showed my homepage title and description, not the article's. Every single post had the same preview because I never told platforms what each article was about.

The content was unique, but to Twitter, LinkedIn, and Google, every post looked identical. I had built the reading experience but ignored how content gets discovered and shared.

---

## How Link Previews Work

When you paste a URL into Twitter or LinkedIn, the platform fetches your page and looks for specific meta tags in the HTML head. It doesn't read your content. It reads metadata.

If those tags exist, you get a proper preview with title, description, and image. If not, the platform guesses, and usually gets it wrong.

### OpenGraph

Facebook created OpenGraph in 2010 to solve this. You add meta tags that describe your page:

```html
<meta property="og:title" content="The Skill That Writes Itself" />
<meta property="og:description" content="An orphan article..." />
<meta property="og:type" content="article" />
<meta property="og:image" content="https://sumitsute.com/images/post.jpg" />
```

Most social platforms read these tags to build preview cards. Twitter has its own format called Twitter Cards, but it falls back to OpenGraph if Twitter-specific tags are missing.

I decided not to use a fallback image. If a post has no image, the preview has no image. Using my homepage screenshot as a default would be misleading.

## How Search Engines Understand Content

Social platforms use OpenGraph. Search engines want structured data in a different format.

### JSON vs JSON-LD

Regular JSON is just a data format. You can structure it however you want:

```json
{
  "title": "My Post",
  "author": "Sumit Sute"
}
```

This is valid JSON, but a search engine doesn't know what it represents. Is it a blog post? A product? An event?

JSON-LD adds a shared vocabulary through the `@context` field:

```json
{
  "@context": "https://schema.org",
  "@type": "BlogPosting",
  "headline": "My Post",
  "author": {
    "@type": "Person",
    "name": "Sumit Sute"
  }
}
```

The `@context` points to schema.org, which defines standard types for web content. The `@type` says this is a BlogPosting. Now search engines understand what they're looking at.

### What is schema.org?

Schema.org is a vocabulary created by Google, Microsoft, Yahoo, and Yandex in 2011. It defines types for things on the web: Person, Organization, BlogPosting, Product, Event, Recipe, and hundreds more.

When you use schema.org types, search engines can show rich results. Recipe cards with cooking times, event listings with dates, product pages with review stars. All of that comes from structured data.

You include JSON-LD as a script tag in your page:

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "BlogPosting",
  ...
}
</script>
```

## Canonical URLs

There's one more piece. When someone links to your post with query parameters like `?utm_source=twitter`, search engines see a different URL. If scrapers copy your content, they create more URLs with identical content. Which one is the original?

```html
<link rel="canonical" href="https://sumitsute.com/bloq/my-post" />
```

This tells search engines which URL is authoritative. It consolidates ranking signals and helps avoid duplicate content issues.

## How the Data Flows

Here's how my bloq posts now get their metadata:

```text
MDX frontmatter (title, summary, tags, image)
    ↓
generateMetadata() reads frontmatter at build time
    ↓
Returns Metadata object with OpenGraph and Twitter tags
    ↓
Next.js injects meta tags into <head>
    ↓
BlogPostingJsonLd component renders schema.org JSON-LD
    ↓
When shared: platforms read meta tags, build correct preview
    ↓
When crawled: search engines read JSON-LD, understand content type
```

The frontmatter I write once in the MDX file becomes the metadata that platforms see everywhere.

---

## What I Actually Built

1. **generateMetadata function** in the bloq page component. Reads post frontmatter and returns title, description, canonical URL, OpenGraph tags, and Twitter card tags.

2. **Conditional images**. If a post has an image, include it in the metadata. If not, leave it empty.

3. **JSON-LD BlogPosting component**. Renders a script tag with headline, author, publication date, and keywords in schema.org format.

4. **Static generation**. All posts pre-rendered at build time, so search engines get complete HTML immediately.

---

## What I Learned

Metadata is about translation. You're teaching platforms how to describe your content in their format.

OpenGraph handles social previews. JSON-LD handles search understanding. Canonical URLs handle content ownership. They serve different purposes but work together.

The implementation wasn't complicated once I understood what each format was for. The hard part was realizing I had ignored this entirely.
