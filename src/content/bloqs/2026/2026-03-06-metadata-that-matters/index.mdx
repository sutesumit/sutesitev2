## The Orphan Problem, Again

> "A page without metadata is a stranger at its own party."

I noticed something embarrassing. Every bloq post on my site — when shared on Twitter, LinkedIn, or indexed by search engines — showed the same thing:

```
Title: sumit sute
Description: Sumit Sute's personal dev page...
```

Not the article title. Not the article summary. Just the generic homepage metadata, copy-pasted across every URL like a lazy form letter.

The content was unique. The code was thoughtful. But to the outside world, every post looked identical. The Skill That Writes Itself? Same metadata as Hello World. Architecture Was Already There? Indistinguishable from my homepage.

I'd built the reading experience. I'd ignored the discovery experience.

---

## What Was Missing

The root `layout.tsx` had metadata, but it was static:

```typescript
export const metadata: Metadata = {
  title: "sumit sute",
  description: "Sumit Sute's personal dev page...",
  // ...openGraph, twitter, etc.
};
```

This is fine for the homepage. But Next.js expects you to generate per-page metadata using `generateMetadata`. I hadn't done that for the bloq routes.

The result: every `/bloq/[slug]` page inherited the root metadata. No unique titles. No article-specific OpenGraph images. No JSON-LD structured data for BlogPosting schema.

---

## The Fix, In Three Parts

### 1. Enhanced Root Metadata

First, I improved the root layout to set a better foundation:

```typescript
export const metadata: Metadata = {
  title: {
    default: 'sumit sute',
    template: '%s | sumit sute',  // Pages use this template
  },
  alternates: {
    canonical: SITE_URL,
  },
  formatDetection: {
    email: false,
    address: false,
    telephone: false,
  },
  openGraph: {
    locale: 'en_US',
    // ...
  },
};
```

The `title.template` means any page that sets `title: "My Post"` becomes `"My Post | sumit sute"`.

I also added JSON-LD structured data for the site itself:

```typescript
const jsonLdWebSite = {
  '@context': 'https://schema.org',
  '@type': 'WebSite',
  name: SITE_NAME,
  url: SITE_URL,
  author: {
    '@type': 'Person',
    name: 'Sumit Sute',
  },
};
```

This gets injected into `<head>` as a script tag. Search engines love it.

---

### 2. Per-Post Metadata with generateMetadata

The real fix was in `src/app/bloq/[slug]/page.tsx`. I added a `generateMetadata` function:

```typescript
export async function generateMetadata({ params }): Promise<Metadata> {
  const { slug } = await params;
  const post = getBloqPostBySlug(slug);

  if (!post) {
    return { title: 'Post Not Found' };
  }

  const postUrl = `${SITE_URL}/bloq/${post.url}`;
  const imageUrl = post.image 
    ? `${SITE_URL}${post.image}` 
    : `${SITE_URL}/sumit-sute-homepage.jpg`;

  return {
    title: post.title,
    description: post.summary,
    alternates: {
      canonical: postUrl,
    },
    openGraph: {
      title: post.title,
      description: post.summary,
      url: postUrl,
      type: 'article',
      publishedTime: post.publishedAt,
      modifiedTime: post.updatedAt,
      tags: post.tags,
      images: [{ url: imageUrl, width: 800, height: 600, alt: post.title }],
    },
    twitter: {
      card: 'summary_large_image',
      title: post.title,
      description: post.summary,
      images: [imageUrl],
    },
  };
}
```

Now each post has:
- Unique title and description
- Canonical URL (important for SEO — tells search engines "this is the real URL")
- OpenGraph with `type: 'article'`, publication date, tags
- Twitter card with the post's image (or fallback to site default)

### 3. BlogPosting JSON-LD Schema

Structured data is how you speak robot to search engines. I added a component that generates BlogPosting schema for each article:

```typescript
function BlogPostingJsonLd({ post }) {
  const jsonLd = {
    '@context': 'https://schema.org',
    '@type': 'BlogPosting',
    headline: post.title,
    description: post.summary,
    datePublished: post.publishedAt,
    dateModified: post.updatedAt || post.publishedAt,
    author: {
      '@type': 'Person',
      name: post.authors[0] || 'Sumit Sute',
    },
    keywords: post.tags.join(', '),
  };

  return (
    <script
      type="application/ld+json"
      dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
    />
  );
}
```

This renders inside the article. Google can now understand: this is a blog post, here's when it was published, here's who wrote it, here's what it's about.

Rich results in search. Better social previews. Proper attribution.

## Static Generation for Free

While I was at it, I added `generateStaticParams`:

```typescript
export async function generateStaticParams() {
  const posts = getBloqPosts();
  return posts.map((post) => ({
    slug: post.url,
  }));
}
```

This tells Next.js: at build time, read all the bloq posts and pre-render them as static HTML. No server-side rendering needed. Fast page loads. Better SEO (crawlers get immediate content).

The build output confirmed it:

```
├ ● /bloq/[slug]
├   ├ /bloq/the-skill-that-writes-itself
├   ├ /bloq/architecture-was-already-there
├   └ [+6 more paths]
```

All nine posts, statically generated at build time.

---

## What Happens When I Add a New Post?

The `generateStaticParams` function calls `getBloqPosts()` at build time. That function reads the filesystem. So:

- **Dev server**: New posts work immediately — metadata generates on-demand
- **Production**: Run `npm run build` and the new post gets pre-rendered

No manual registration. No database to update. Drop a file in `src/content/bloqs/`, rebuild, done.

---

## What I Took Away

- **Metadata is part of the content.** If you write something worth reading, make sure it introduces itself properly when shared.
- **generateMetadata is not optional for dynamic routes.** Without it, every page inherits the layout's metadata.
- **JSON-LD is underused.** It's a few lines of code that unlock rich search results. Worth the effort.
- **Static generation is a nice side effect.** Once you have `generateMetadata`, adding `generateStaticParams` is trivial.
- **Canonical URLs matter.** If someone copies your content, search engines know which version is the original.

The bloq now speaks for itself. Each article has its own voice, its own preview, its own identity in the machines that distribute it.

The metadata writes itself. I just had to ask.
