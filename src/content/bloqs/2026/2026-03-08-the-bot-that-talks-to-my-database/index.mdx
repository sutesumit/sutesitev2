## What's a Blip?

> "Every UUID slice is a cry for help."

[Blips](/blip) are short, 280-character posts. Think of them as tweets that live on my own infrastructure. They're stored in Supabase, rendered on a Next.js page, and until recently, identified by UUIDs.

The UUID problem wasn't theoretical. Somewhere in the UI, I was displaying the first two characters of a UUID like that meant something. `a3`, `f7`, `2b`. Meaningless placeholders.

The goal: serial numbers you could reference, type, and remember.

---

## The Feature at a Glance

Two pieces:

1. **Sequential base-62 serials** for each blip (`0, 1, 2... 9, a, b... Z, 10`)
2. **Telegram bot** to create, read, update, and delete blips using those serials

The flow:

```text
┌─────────────┐     ┌──────────────┐     ┌──────────┐
│  Telegram   │────▶│  Next.js API │────▶│ Supabase │
│  Message    │     │  Routes      │     │ Database │
└─────────────┘     └──────────────┘     └──────────┘
                           │
                           ▼
                    blip_serial (base-62)
```

User sends a message. Bot processes it. API queries by serial. Response flows back.

## Concept 1: Base-62 Encoding

Base-62 uses all alphanumeric characters: `0-9`, `a-z`, `A-Z`. That's 62 symbols.

| Decimal | Base-62 |
|---------|---------|
| 0       | 0       |
| 10      | a       |
| 35      | z       |
| 61      | Z       |
| 62      | 10      |
| 3844    | 100     |

Why base-62?

- **URL-safe** — no special characters to encode
- **Case-sensitive** — 62 symbols instead of 36
- **Compact** — four digits in base-62 covers 14 million values
- **Human-readable** — `a3` is easier to remember than `a3f2e8c1-...`

The encoding is reversible. Given `a3`, you can decode back to the original number. This means the serial is both identifier and display value, no lookup table needed.

---

## Concept 2: Database-Generated Serials

The serial isn't generated in application code. PostgreSQL handles it.

The mechanism:

1. A **sequence** tracks the next number (standard Postgres feature)
2. A **function** converts the number to base-62 (custom PL/pgSQL)
3. A **trigger** fires on every INSERT, calling the function

This is atomic. No race conditions, no duplicate checks, no application-level coordination. The database guarantees uniqueness because sequences are designed for exactly this.

### The Backfill Problem

Existing blips needed serials too. The solution: a migration that:

1. Orders existing blips by creation date
2. Assigns sequential numbers starting from 0
3. Converts each number to base-62
4. Updates the `blip_serial` column

Oldest blip gets `0`. Next gets `1`. Chronological order preserved.

The tricky part: PostgreSQL's `substr()` function expects INTEGER arguments, but sequences return BIGINT. A simple cast (`::INTEGER`) fixed it, but not before I questioned whether base-10 was "humble enough."

## Concept 3: Dynamic Routes by Serial

Next.js App Router supports dynamic segments in routes. Instead of querying by UUID:

```text
GET /api/blip/a3f2e8c1-4b5d-...
```

You query by serial:

```text
GET /api/blip/a3
```

The route handler extracts the serial from the URL, queries Supabase with `.eq("blip_serial", serial)`, and returns the blip.

Same pattern for PUT (update) and DELETE. The serial becomes the primary lookup key for all operations. The UUID still exists internally, but the public interface uses serials.

## Concept 4: Grammy Bot and Middleware Order

The Telegram bot uses [grammy](https://grammy.dev/), a TypeScript-first bot framework.

The key insight: **middleware order determines handler execution**.

Handlers are registered in code. When a message arrives, grammy checks each handler in order. The first match wins.

This means:

- **Specific handlers first** — commands like `/start`, `/list`, `/edit`
- **Generic handlers last** — catch-all for regular messages

If you register the generic handler first, it matches everything. Commands never get a chance. The bot receives the message but doesn't respond correctly.

I learned this the hard way. The bot was running, polling, receiving messages. But `/start` returned nothing. The message handler was catching everything, returning early for commands, and the command handlers never ran.

Order matters. Middleware is a pipeline, not a buffet.

### Session State for Conversational Flows

The `/edit` command needs to remember context across messages:

```text
User: /edit a3
Bot: Editing blip a3. Send new content:
User: Updated text here
Bot: Updated blip a3
```

Grammy's session middleware stores state between messages. The bot remembers `mode: "edit"` and `editingSerial: "a3"` until the operation completes or is cancelled.

Without sessions, every message is stateless. The bot can't ask follow-up questions.

## Concept 5: Dual Auth Systems

Two authentication mechanisms coexist:

| System | Used By | Mechanism |
|--------|---------|-----------|
| Secret key | curl, direct API | Header-based token |
| User ID whitelist | Telegram bot | Numeric ID check |

They don't overlap. The bot doesn't know about the API key. The API routes don't know about Telegram user IDs. Each system trusts its own source.

For a personal tool, this is sufficient. No JWTs, no OAuth, no shared token store. Just two simple checks in two different places.

The Telegram user ID whitelist is particularly simple: a comma-separated list of numbers in an environment variable. If your ID is in the list, you're authorized. If not, you get "Unauthorized."

## The Full Flow: Creating a Blip

When you send "Hello world" to the bot:

1. **Telegram receives** the message on their servers
2. **Bot polls** Telegram's API for new updates (long polling)
3. **Grammy parses** the update and matches handlers
4. **Auth check** — is this user ID in the whitelist?
5. **Mode check** — is there an active edit/delete session?
6. **Command check** — does the text start with `/`?
7. **Create blip** — INSERT into Supabase
8. **Trigger fires** — generates base-62 serial atomically
9. **Response** — bot replies with the new serial

The user sees: `Created blip a3`

---

## The Full Flow: Reading a Blip

When you send `/get a3`:

1. **Telegram receives** the message
2. **Bot polls** and grammy parses
3. **Command matched** — `/get` handler runs (registered before generic handler)
4. **Serial extracted** from the command argument
5. **Query** — SELECT WHERE blip_serial = 'a3'
6. **Response** — bot replies with content

The user sees: `a3. Hello world`

---

## What I Took Away

- **Middleware order is execution order.** Register specific handlers before generic ones. The pipeline doesn't reroute itself.
- **Database-generated serials are atomic.** No application-level coordination, no collision detection, no retry logic. The sequence handles it.
- **Base-62 is compact and reversible.** The serial is both identifier and display value.
- **Dual auth is fine for personal tools.** API key for direct access, user ID whitelist for Telegram. No need to unify them.
- **Sessions enable conversational flows.** Edit mode persists across messages. The bot remembers context.

```text
└── blip.ts             # Supabase queries

src/app/api/
├── blip/route.ts       # Create and list
├── blip/[serial]/route.ts  # Read, update, delete by serial

└── 001_add_blip_serial.sql  # Column, sequence, trigger, backfill
```
