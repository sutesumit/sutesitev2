## Why a Telegram Bot?

> "The best interface is the one you already have open."

[Blips](/blip) are short, 280-character posts. Think of them as tweets, except they live on my server, and nobody can quote-tweet them with a bad take.

Until now, posting a blip meant opening a terminal, crafting a curl command, and hoping I remembered the API key. This is fine if you're a robot. I am not a robot. I am a human who wants to type words into a phone while walking.

Telegram bots are perfect for this. They live in the same app where I already chat with humans. No separate app, no OAuth dance, no "confirm your identity" emails. Just a conversation with a very obedient piece of software.

---

## How Telegram Bots Work

### The Bot Father

Every bot starts with [@BotFather](https://t.me/botfather). It's exactly what it sounds like: a bot that makes bots.

Send `/newbot`, pick a name, get a token. That token is your bot's identity. Lose it and your bot becomes an orphan. Give it to someone and they become the new parent. Bot custody law is simple: possession is everything.

With this token, you can call Telegram's API. `sendMessage` to reply. `getMe` to ask "who am I?" (philosophical, but it returns JSON). The API is just HTTP. No SDK required, though frameworks help.

### Bots vs Users

Bots are special Telegram accounts with restrictions:

- They can't start conversations (users must message first, like a shy friend)
- They have rate limits (no spamming)
- They receive commands (messages starting with `/`)

The important part: bots don't have passwords. If you have the token, you control the bot. This is either terrifying or liberating, depending on your relationship with secrets.

## Getting Messages: The Two Approaches

Here's the core problem: someone sends a message to your bot on Telegram. How does your code find out?

### Imagine Life Without Either

You could write a script that logs into Telegram's website, scrapes for new messages, copies them somewhere. Every few minutes. Forever. Like checking the mail but the mail is JSON and the post office is a server farm in Singapore.

That's obviously terrible. Telegram solved this problem two ways, and both are better than my hypothetical nightmare.

### Polling: The "Check Your Mailbox" Approach

Polling is like walking to your mailbox every 30 seconds. Most of the time, it's empty. But you keep checking, because maybe this time there's mail.

Your bot repeatedly asks Telegram: "Any new messages?"

```text
Bot: Any messages?
Telegram: No.
(wait)
Bot: Any messages?
Telegram: No.
(wait)
Bot: Any messages?
Telegram: YES! Finally! Here's "hello world" from user 123.
Bot: (processes, replies)
Bot: Any messages?
Telegram: No.
(wait)
Bot: Any messages?
Telegram: I already said no.
```

**The good:** It works anywhere. Laptop, server, Raspberry Pi in a closet. No configuration. Just run the script and it starts asking questions.

**The bad:** You're making API calls constantly, even when nobody loves your bot enough to message it. There's a small delay. The script needs to run forever, like a digital pet that never sleeps.

### Webhooks: The "Mailman Knocks" Approach

Webhooks flip the relationship. Instead of you asking Telegram, Telegram tells you.

You give Telegram a URL. When a message arrives, Telegram POSTs to that URL.

```text
User sends message → Telegram servers → POST to your URL → Your code wakes up
```

It's like giving the mailman your phone number. When mail arrives, they text you. You don't check the mailbox. You live your life until the notification comes.

**The good:** Real-time. No wasted calls. Your server sleeps until there's work.

**The bad:** You need a public HTTPS URL. Telegram can't POST to `localhost:3000`. Your laptop is behind a router, behind a firewall, behind several layers of networking equipment you don't understand.

### What's ngrok?

[ngrok](https://ngrok.com/) is a tunnel. It punches through your firewall and creates a public URL that forwards to your laptop.

```text
Public URL (https://abc123.ngrok.io)
         │
         ▼
    ngrok servers (the tunnel entrance)
         │
         ▼
    your firewall (ngrok goes around it)
         │
         ▼
    your laptop (localhost:3000)
```

You run `ngrok http 3000`, and suddenly your local machine has a public address. It's like giving your apartment a ZIP code.

For development, this means webhooks work locally. For production, you'd deploy somewhere with a real domain. Or keep using ngrok and explain to visitors why your URL changes every time you restart the tunnel.

### Which I Used

Both. The bot supports polling (`npm run bot`) and webhooks (via a Next.js API route).

For development, polling is easier. Run the script, done. For production, webhooks are cleaner. The bot only wakes up when there's work, like a consultant who bills by the hour.

## Grammy: Making Sense of Bot Code

You could call Telegram's API directly. Raw HTTP requests. Parse JSON yourself. Handle errors manually. Build character through suffering.

Or you could use [Grammy](https://grammy.dev/), a framework that handles the plumbing. Think of it as a translator between Telegram's raw data and your actual logic.

### Handlers: "When X happens, do Y"

A handler is a function that runs when something specific happens.

```text
User sends /start → start handler runs
User sends /list → list handler runs
User sends "hello" → message handler runs
```

Without a framework, you'd write a giant if-else chain. "If message starts with /start, do this. Else if it starts with /list, do that. Else if..." Grammy lets you register handlers and handles the matching for you.

### Middleware: The Assembly Line

Imagine a factory. A product moves from station to station. Station 1 checks quality. Station 2 adds a label. Station 3 packages it. Each station does one thing and passes it along.

Middleware works the same way. An incoming message passes through a chain of functions:

```text
Message arrives
    │
    ▼
Auth check: Is this user allowed?
    │ (yes, proceed)
    ▼
Session check: Are we mid-conversation?
    │ (no, proceed)
    ▼
Command check: Is this /start? /list? /edit?
    │ (no match)
    ▼
Message handler: Process as regular message
```

Each station can process and pass along, or stop the chain early. If auth fails, no need to check commands. The message dies there.

### The Order Problem

Here's what bit me: assembly line order matters.

I registered the generic "message handler" before the specific "command handlers." Every message hit the message handler first. Commands like `/start` were processed as regular messages. The command handlers never ran.

The bot was running. It received messages. It just... did nothing useful with them. Like hiring a receptionist who answers every question with "I don't know, try asking someone else" but there's no one else.

The fix: specific before generic.

```text
Wrong:
1. Is it a message? Yes → handle it (commands never reach step 2)
2. Is it /start? (dead code)

Right:
1. Is it /start? Yes → handle it
2. Is it /list? Yes → handle it
3. Is it a message? Yes → handle it (catch-all)
```

This isn't grammy-specific. Any middleware system works this way. But it's the kind of thing you learn by breaking it first and wondering why your bot is so quiet.

## The Edit Command: Two Ways to Do It

The `/edit` command has an identity crisis. It supports two modes.

### Mode 1: One and Done

```text
User: /edit a3 This is the new content
Bot: Updated blip a3
```

Everything in one message. Serial and new content together. No session state needed. The bot parses the serial, finds the space, takes everything after as new content. Done.

### Mode 2: Conversational

```text
User: /edit a3
Bot: Editing blip a3:
     "This is the old content"
     
     Send new content:
User: This is the new content
Bot: Updated blip a3
```

Two messages. The bot needs to remember "we're editing a3" between them. That's session state.

### Why Support Both?

I built conversational mode first. It felt more natural. You see what you're editing before committing.

Then I realized: sometimes you already know what you want to write. Typing `/edit a3 new text` is faster than a back-and-forth. Why force the conversation?

So now it does both. If you include content after the serial, it's one-and-done. If you just send the serial, it shows the current content and waits. The bot adapts to how you want to use it.

This is a small thing, but it matters. The bot doesn't impose a workflow. It supports yours.

## Authentication: The Simplest Thing That Works

Telegram sends the user's ID with every message. It's a number. Permanent, unique, can't be faked.

The bot checks this ID against a whitelist:

```text
TELEGRAM_ALLOWED_USER_IDS=123456789,987654321
```

If your ID is in the list, you're in. If not, "Unauthorized."

No JWTs. No OAuth. No "sign in with Google." No password resets. No forgot-your-password emails. Just: is this number in the list?

For a personal bot, this is perfect. The security model is: if you have my phone, you can post blips. If you don't have my phone, you have bigger problems than my bot.

## The Commands

| Command | What it does |
|---------|--------------|
| `/start` | Show help (the bot says hello) |
| `/list` | Show recent blips |
| `/get a3` | Fetch blip a3 |
| `/edit a3 new text` | Update blip a3 (one-shot) |
| `/edit a3` | Enter edit mode (conversational) |
| `/del a3` | Delete blip a3 (no confirmation, live dangerously) |
| (any text) | Create a new blip |

Most are instant. Only `/edit` without content enters conversational mode.

## The Full Flow: A Message Travels

When you send "Hello world" to the bot:

```text
1. You type "Hello world" in Telegram
2. Message travels to Telegram's servers (probably Singapore)
3. Telegram either:
   - Holds it for polling (bot will ask soon), OR
   - POSTs to webhook immediately
4. Bot receives the message
5. Bot runs the assembly line:
   - Auth: Is user allowed? Yes.
   - Session: In edit mode? No.
   - Commands: Is it /start? /list? No.
   - Message handler: Create blip.
6. Bot replies: "Created blip a3"
7. Reply travels back to your phone
```

Usually under a second. Most of that is light traveling through fiber optic cables. The bot's logic is milliseconds.

## What I Took Away

- **Polling for dev, webhooks for prod.** Polling works anywhere. Webhooks scale better but need a public URL.
- **Middleware is an assembly line.** Order matters. Specific before generic. Learn by breaking.
- **Support multiple workflows.** One-shot `/edit a3 text` and conversational `/edit a3` both work. Let users choose.
- **User ID is free authentication.** Telegram sends it. Use it. Keep your JWTs elsewhere.
- **ngrok is magic.** A tunnel from the public internet to your laptop. Essential for webhook development.
