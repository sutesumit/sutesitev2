## The Thing I Built

> "The best interface is the one you already have open."

So I have these [blips](/blip). Short posts, 280 characters. Like tweets, but mine.

Posting one used to mean curl. Terminal. API key. Fine for a robot, annoying for a human on a phone.

Now there are two interfaces. Terminal commands for when I'm working. Telegram bot for when I'm not. Same database, same blips, different entry points depending on where I am.

This article is about the Telegram bot. How it works, what I learned building it.

---

## Telegram Bots, Quickly

You talk to @BotFather on Telegram. Say `/newbot`, pick a name, get a token. That token is your bot's password. Lose it and the bot is orphaned.

With the token, you call Telegram's API. `sendMessage` to reply. `getUpdates` to fetch messages. It's HTTP. JSON in, JSON out. Nothing fancy.

Bots are weird accounts. They can't start conversations. They have rate limits. They get commands (stuff starting with `/`). But mostly: they're just code that talks to Telegram's servers.

## Polling vs Webhooks

Okay so someone messages your bot. How does your code know?

Two ways.

### Polling

Your code keeps asking: "Any new messages?"

```text
You: Anything?
Telegram: No.
(wait)
You: Anything?
Telegram: No.
(wait)
You: Anything?
Telegram: Yes! User 123 said "hello".
```

It works. It's simple. The downside is you're making calls constantly, even when nobody's messaging you. And there's a small delay between when the message arrives and when you ask for it.

### Webhooks

You give Telegram a URL. When a message arrives, Telegram POSTs to that URL.

```text
User sends message → Telegram → POST to your URL → You handle it
```

No polling. No wasted calls. Real-time.

But: your URL needs to be public. Telegram can't POST to `localhost`. So you need a real server with HTTPS.

### What I Use

Webhooks in production. The bot lives at `/api/telegram/webhook`. Telegram POSTs there, the handler does its thing.

### Dev Setup

Polling exists for local testing. Run `npm run bot` and it starts polling Telegram's API. Useful for development before you have a production server with HTTPS.

I don't use it anymore. But it's there if I need to debug something locally.

**Why keep it?** Zero-config testing. No ngrok, no tunnel, no public URL. Just run the script.

**Why remove it?** Dead code if you never use it. Adds dotenv as a dependency. Confuses the architecture (two ways to do the same thing).

For now, it stays. Might not forever.

## Grammy and Middleware

I use [grammy](https://grammy.dev/) for the bot. It's a TypeScript framework. Handles the API stuff so I don't have to.

The interesting bit is middleware.

### What Middleware Actually Is

You know how sometimes you have a function, and you want to run some code before it? Like, check if the user is logged in before running the handler?

You could do:

```text
handler() {
  if (!loggedIn) return
  // actual logic
}
```

But now every handler needs that check. And maybe you also want logging. And error handling. And rate limiting. Suddenly every handler starts with the same ten lines.

Middleware is a pattern for this. Instead of one big function, you have a chain of small ones. Each does one thing and passes control to the next.

```text
request → auth → logging → handler → response
```

In grammy, you register handlers in order. When a message comes in, it goes through each one. If a handler matches, it runs. If it calls `next()`, the next handler runs too. If it doesn't, the chain stops.

### The Bug I Hit

I registered handlers in the wrong order.

```text
message handler (catches everything)
command handlers (never reached)
```

The message handler caught `/start` before the command handler could see it. Bot received messages, did nothing useful.

Fixed it by putting commands first:

```text
/start handler
/list handler
/edit handler
message handler (catch-all at the end)
```

Order matters. Specific before generic. Obvious in hindsight.

## No Sessions

The bot doesn't use sessions.

Originally I thought `/edit` would be conversational. You'd type `/edit a3`, bot shows current content, you type the new text. Two messages. State between them.

Then I thought: why? Just do `/edit a3 new text` in one message. Parse the serial, take everything after as content. Done.

No session middleware. No state to manage. No "cancel" command for abandoned edits. The bot is stateless. Each message is independent.

This is simpler. Whether it's better depends on what you're building. For a CRUD bot, stateless works.

## Auth

Telegram sends the user's ID with every message. Can't be faked.

I check it against a whitelist:

```text
TELEGRAM_ALLOWED_USER_IDS=123456789
```

If your ID matches, you're in. If not, "Unauthorized."

No JWT. No OAuth. No login flow. Just: is this number in the list?

For a personal bot, this is the right amount of security. If you have my phone, you can post blips. If you don't, you can't.

## The Commands

| Command | What it does |
|---------|--------------|
| `/start` | Help text |
| `/list` | Recent blips |
| `/get a3` | Show blip a3 |
| `/edit a3 new text` | Update blip a3 |
| `/del a3` | Delete blip a3 |
| any text | Create new blip |

All single-message. No conversations. Stateless.

## What I Learned

**Webhooks over polling for production.** Polling works, but webhooks are cleaner. Your server only wakes up when there's work.

**Middleware order matters.** Register specific handlers before generic ones. The chain runs in order. Learn this by breaking it once.

**Stateless is simpler.** Sessions add complexity. If you can avoid them, avoid them. `/edit a3 text` is one message. No state needed.

**User ID is free auth.** Telegram sends it. Use it. Keep your JWTs for something else.

## Two Interfaces

There are two ways to post a blip now.

**Terminal.** Curl commands hitting the API directly. `blip "content=thought"` creates one. `blips` lists them. Fast when I'm already in a terminal, which is most of the time when working.

**Telegram.** The bot. Open the app, type, done. Fast when I'm not at a computer. Walking, in bed, at a cafe.

Both hit the same database. Both are instant. No switching contexts, no waiting, no friction.

The curl commands live in a separate file. Shell aliases. `blip`, `blips`, `blip-get`, `blip-edit`, `blip-del`. Not in this article because they're not interesting. Just HTTP calls with an auth header.

---

## Why This Matters

The whole point: I can post a thought the moment I form it.

Before: idea strikes → remember it → find terminal → craft curl command → post. By step 3, the idea might be gone. Or diluted. Or I'm doing something else.

Now: idea strikes → open Telegram (or terminal) → type → done. Three seconds. The thought is captured before I can second-guess it.

This is what good tools do. They get out of the way. The bot isn't the point. The blips aren't the point. Capturing thoughts while they're fresh is the point. The interfaces just make that frictionless.

## A Note on Security

Both interfaces require auth. The API uses a secret key in a header. The bot uses a whitelist of Telegram user IDs.

The security model is simple: if you have my phone or my laptop, you can post blips. If you have neither, you can't.
