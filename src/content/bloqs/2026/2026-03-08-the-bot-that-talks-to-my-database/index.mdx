## Why a Telegram Bot?

> "The best interface is the one you already have open."

[Blips](/blip) are short, 280-character posts. Think of them as tweets on my own infrastructure. Until now, creating one meant opening a terminal, crafting a curl command, and hoping I remembered the API key.

I wanted something simpler. Open Telegram. Type a message. Done.

Telegram bots are perfect for this. They live in the same app where I already chat with humans. No separate app, no authentication flow, no context switching. Just a conversation.

---

## How Telegram Bots Work

Telegram's Bot API is HTTP-based and surprisingly simple.

### The Bot Father

Every bot starts with [@BotFather](https://t.me/botfather). Send `/newbot`, pick a name, get a token. That token is your bot's identity. It looks like:

```text
123456789:ABCdefGHIjklMNOpqrsTUVwxyz
```

With this token, you can call any Bot API endpoint:

- `getMe` — get bot info
- `getUpdates` — fetch new messages
- `sendMessage` — reply to a user
- `setWebhook` — register a webhook URL

### Bots vs Users

Bots are special Telegram accounts:

- They can't initiate conversations (users must start first)
- They have a limited rate of API calls
- They can be added to groups
- They receive commands (messages starting with `/`)

The important part: bots don't have passwords. Authentication is purely token-based. If you have the token, you control the bot.

## Getting Messages: Polling vs Webhooks

There are two ways to receive messages from Telegram.

### Polling

Your bot repeatedly calls `getUpdates` to fetch new messages:

```text
Bot: Any new messages?
Telegram: Here's a list.
Bot: (processes messages)
Bot: Any new messages?
Telegram: Nothing new.
(repeat forever)
```

**Pros:**
- Simple to implement
- Works behind NAT/firewall
- No public URL needed

**Cons:**
- Constant API calls (even when idle)
- Slight delay between message and response
- Needs a running process

### Webhooks

You register a public HTTPS URL with Telegram. When a message arrives, Telegram POSTs it to your URL:

```text
User sends message → Telegram servers → POST to your URL → Your server responds
```

**Pros:**
- Real-time delivery
- No wasted API calls
- Scales better

**Cons:**
- Requires public HTTPS URL
- Needs TLS certificate
- More complex setup (ngrok for local dev)

### Which I Used

Both. The bot supports polling for development (`npm run bot`) and webhooks for production (via a Next.js API route). The webhook endpoint at `/api/telegram/webhook` receives POST requests and passes them to the same handler logic.

For local development, polling is easier. No ngrok, no tunnel, just run the script. For production, webhooks are cleaner. The bot only wakes up when there's actual work.

## Grammy: A TypeScript-First Bot Framework

[Grammy](https://grammy.dev/) is a bot framework designed for TypeScript. It handles the low-level API calls and provides structure for handling messages.

### Handlers and Middleware

Grammy uses a middleware pattern. You register handlers, and when an update arrives, each handler gets a chance to process it:

```text
Update arrives → Handler 1 → Handler 2 → Handler 3 → Response
```

Handlers can:
- Process the update and respond
- Pass to the next handler
- Stop the chain

### The Order Problem

Here's what bit me: **handlers run in registration order**.

I registered a generic "message" handler first, then specific "command" handlers. The message handler caught everything, including commands. My `/start` command never ran because the message handler already processed it.

The fix: register specific handlers before generic ones.

```text
Commands first  → /start, /list, /edit
Generic last    → catch-all for regular messages
```

Middleware is a pipeline. Order matters. This isn't unique to grammy, it's how most bot frameworks work. But it's the kind of thing you learn by breaking it first.

## Session State for Conversational Flows

Some operations need multiple messages. The `/edit` flow:

```text
User: /edit a3
Bot: Editing blip a3. Send new content:
User: Updated text here
Bot: Updated blip a3
```

The bot needs to remember: "we're in edit mode, targeting blip a3."

Grammy's session middleware stores state per chat. When the first message arrives, the bot sets `mode: "edit"` and `target: "a3"`. When the second message arrives, the bot checks the session, sees it's in edit mode, and processes accordingly.

Without sessions, every message is independent. The bot can't ask follow-up questions or maintain context.

## Authentication: User ID Whitelist

Telegram sends the user's ID with every message. It's a number, permanent and unique.

The bot checks this ID against a whitelist stored in an environment variable:

```text
TELEGRAM_ALLOWED_USER_IDS=123456789,987654321
```

If the ID is in the list, the request proceeds. If not, "Unauthorized."

This is deliberately simple. No JWTs, no OAuth, no token refresh. For a personal bot, a hardcoded whitelist is sufficient. The security model is: if you have my phone, you have access to the bot. If you don't, you don't.

## The Commands

The bot supports these commands:

| Command | What it does |
|---------|--------------|
| `/start` | Show help |
| `/list` | Show recent blips |
| `/get <id>` | Fetch a specific blip |
| `/edit <id>` | Edit a blip (conversational) |
| `/del <id>` | Delete a blip |
| (any text) | Create a new blip |

The conversational edit flow uses sessions. The delete and create flows are single-message. The list and get commands are read-only.

## The Full Flow: A Message Travels

When you send "Hello world" to the bot:

```text
1. You type "Hello world" in Telegram
2. Telegram's servers receive the message
3. Either:
   a. Bot polls getUpdates → receives message, OR
   b. Telegram POSTs to webhook URL
4. Grammy parses the update
5. Grammy runs handlers in order:
   - Is it /start? No.
   - Is it /list? No.
   - Is it /edit? No.
   - Is it a regular message? Yes.
6. Handler checks: is user ID whitelisted? Yes.
7. Handler checks: are we in edit/delete mode? No.
8. Handler creates blip in database
9. Handler replies: "Created blip a3"
10. Response travels back: bot → Telegram → your phone
```

Latency is usually under a second. Most of that is network round-trip time.

## Webhook Setup for Production

To use webhooks instead of polling:

1. Deploy your bot endpoint (e.g., `https://yoursite.com/api/telegram/webhook`)
2. Register it with Telegram:

```text
POST https://api.telegram.org/bot<TOKEN>/setWebhook
?url=https://yoursite.com/api/telegram/webhook
```

3. Telegram will now POST updates to that URL

For local development, use [ngrok](https://ngrok.com/) or [Cloudflare Tunnel](https://developers.cloudflare.com/cloudflare-one/connections/connect-apps/) to expose localhost:

```text
ngrok http 3000
# → https://abc123.ngrok.io

curl -F "url=https://abc123.ngrok.io/api/telegram/webhook" \
  https://api.telegram.org/bot<TOKEN>/setWebhook
```

## What I Took Away

- **Polling for dev, webhooks for prod.** Polling is zero-setup. Webhooks scale better.
- **Middleware order matters.** Register specific handlers before generic ones.
- **Sessions enable conversations.** Without state, every message is isolated.
- **User ID is free auth.** Telegram sends it with every message. Use it.
- **The Bot API is well-designed.** HTTP-based, simple endpoints, predictable behavior. Rare for a messaging API.
