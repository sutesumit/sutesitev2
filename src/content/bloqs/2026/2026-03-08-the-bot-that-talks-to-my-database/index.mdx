## Why a Telegram Bot?

> "The best interface is the one you already have open."

[Blips](/blip) are short, 280-character posts. Think of them as tweets on my own infrastructure. Until now, creating one meant opening a terminal, crafting a curl command, and hoping I remembered the API key.

I wanted something simpler. Open Telegram. Type a message. Done.

Telegram bots are perfect for this. They live in the same app where I already chat with humans. No separate app, no authentication flow, no context switching. Just a conversation.

---

## How Telegram Bots Work

### The Bot Father

Every bot starts with [@BotFather](https://t.me/botfather). Send `/newbot`, pick a name, get a token. That token is your bot's identity.

With this token, you can call Telegram's API. Want to send a message? Call `sendMessage`. Want to know who your bot is? Call `getMe`. The API is just HTTP requests. No SDK required, though frameworks help.

### Bots vs Users

Bots are special Telegram accounts:

- They can't initiate conversations (users must message first)
- They have rate limits
- They receive commands (messages starting with `/`)

The important part: bots don't have passwords. If you have the token, you control the bot. Keep it secret.

## Getting Messages: The Two Approaches

Here's the core problem: someone sends a message to your bot on Telegram. How does your code find out?

### Imagine Life Without Either

You could write a script that manually checks for messages. Log into Telegram's website, look for new messages, copy them somewhere. Every few minutes. Forever.

That's obviously terrible. Telegram solved this problem two ways.

### Polling: The "Check Your Mailbox" Approach

Polling is like walking to your mailbox every 5 minutes to see if there's mail.

Your bot repeatedly asks Telegram: "Any new messages?"

```text
You: Any messages for me?
Telegram: Nope.
(wait 5 seconds)
You: Any messages for me?
Telegram: Nope.
(wait 5 seconds)
You: Any messages for me?
Telegram: Yes! Here's one from user 123.
You: (process message, send reply)
You: Any messages for me?
Telegram: Nope.
(repeat forever)
```

**The good:** It works anywhere. Your laptop, a server behind a firewall, a Raspberry Pi in your closet. No configuration needed. Just run the script.

**The bad:** You're making API calls constantly, even when nobody is messaging you. There's a small delay (the time between polls). Your script needs to run 24/7.

### Webhooks: The "Mailman Knocks" Approach

Webhooks flip the relationship. Instead of you asking Telegram, Telegram tells you.

You give Telegram a URL. When a message arrives, Telegram sends an HTTP POST request to that URL with the message data.

```text
User sends message → Telegram's servers → POST to your URL → Your code receives it
```

It's like giving the mailman your phone number. When mail arrives, they text you. You don't check the mailbox. You wait for the notification.

**The good:** Real-time. No wasted API calls. Your server only wakes up when there's actual work.

**The bad:** You need a publicly accessible HTTPS URL. Telegram can't POST to `localhost:3000`. Your laptop is behind a router, probably behind a firewall, definitely not reachable from the internet.

### What's ngrok?

This is where [ngrok](https://ngrok.com/) comes in. It creates a tunnel from the public internet to your laptop.

```text
Public URL (https://abc123.ngrok.io)
         │
         ▼
    ngrok servers
         │
         ▼
    tunnel through firewall
         │
         ▼
    your laptop (localhost:3000)
```

You run `ngrok http 3000`, and it gives you a public URL. Requests to that URL flow through ngrok's servers, through the tunnel, to your local machine.

For development, this means webhooks are possible. You get a real HTTPS URL that Telegram can POST to. For production, you'd deploy to a server with a real domain.

### Which I Used

Both. The same bot code supports polling (`npm run bot`) and webhooks (via a Next.js API route).

For development, polling is easier. No ngrok, no tunnel, just run the script. For production, webhooks are cleaner. The bot only wakes up when there's actual work.

## Grammy: Making Sense of Bot Code

You could call Telegram's API directly. Raw HTTP requests. Parse JSON yourself. Handle errors. But that gets tedious fast.

[Grammy](https://grammy.dev/) is a framework that handles the plumbing. Think of it as a translator between Telegram's raw data and your logic.

### Handlers: "When X happens, do Y"

A handler is just a function that runs when something happens.

```text
When user sends /start → run the start handler
When user sends /list → run the list handler
When user sends any message → run the message handler
```

Without a framework, you'd write a giant if-else chain: "if the message starts with /start, do this. If it starts with /list, do that. Otherwise, do this other thing."

Grammy lets you register handlers for specific situations. It handles the matching logic.

### Middleware: The Assembly Line

Imagine an assembly line. A product moves from station to station. Each station does one thing. Maybe station 1 checks quality. Station 2 adds a label. Station 3 packages it.

Middleware works the same way. An incoming message passes through a chain of functions. Each function can:

- Do something and pass it along
- Do something and stop the chain (respond early)
- Modify the message for the next function

```text
Message arrives
    │
    ▼
Auth check: Is this user allowed?
    │ (yes)
    ▼
Session check: Are we in the middle of an operation?
    │ (no)
    ▼
Command check: Is this a /command?
    │ (no)
    ▼
Message handler: Process as regular message
```

### The Order Problem

Here's what bit me: the order of the assembly line matters.

I put "message handler" before "command handler." Every message hit the message handler first. Commands like `/start` were treated as regular messages. The command handlers never ran.

The fix: put specific checks before generic ones.

```text
Wrong order:
1. Is it a message? Yes → process it (commands never reach step 2)
2. Is it /start? (never reached)

Right order:
1. Is it /start? Yes → run start handler
2. Is it /list? Yes → run list handler
3. Is it a message? Yes → process it (catch-all)
```

This isn't grammy-specific. Any system with middleware works this way. But it's the kind of thing you learn by breaking it first.

## Session State: Do We Need It?

The `/edit` command works in two steps:

```text
User: /edit a3
Bot: Editing blip a3. Send new content:
User: Updated text here
Bot: Updated blip a3
```

The bot needs to remember "we're editing blip a3" between messages. That's session state.

### An Alternative: One Message, All The Info

What if `/edit` took everything in one message?

```text
User: /edit a3 Updated text here
Bot: Updated blip a3
```

No session needed. The bot parses the serial and the new content from a single message. Done.

**This would be simpler.** No session middleware, no state to manage, no "cancel" command needed when someone abandons an edit.

So why didn't I do it?

Two reasons:

1. **User experience.** Sometimes you want to see the current content before editing. The two-step flow lets the bot show you what you're editing.

2. **Long content.** If the new text is long, cramming it after the serial gets awkward. `/edit a3 This is a very long piece of text that...`

The conversational flow feels more natural for edits. But for a simpler bot, single-message commands would work fine. It's a tradeoff.

## Authentication: The Simplest Thing That Works

Telegram sends the user's ID with every message. It's a number, permanent and unique to each Telegram account.

The bot checks this ID against a whitelist:

```text
TELEGRAM_ALLOWED_USER_IDS=123456789,987654321
```

If the ID matches, the request proceeds. If not, "Unauthorized."

No JWTs. No OAuth. No login flow. No password reset. Just: "Is this number in the list?"

For a personal bot, this is sufficient. The security model is: if you have my phone, you can post blips. If you don't, you can't.

## The Commands

| Command | What it does |
|---------|--------------|
| `/start` | Show help |
| `/list` | Show recent blips |
| `/get a3` | Fetch blip a3 |
| `/edit a3` | Edit blip a3 (conversational) |
| `/del a3` | Delete blip a3 |
| (any text) | Create a new blip |

Most are single-message. Only `/edit` uses session state.

## The Full Flow: A Message Travels

When you send "Hello world" to the bot:

```text
1. You type "Hello world" in Telegram app
2. Message travels to Telegram's servers
3. Telegram either:
   - Holds it for polling (bot will ask soon), OR
   - POSTs it to the webhook URL immediately
4. Bot receives the message
5. Bot runs through handlers:
   - Is user allowed? Check whitelist. Yes.
   - In edit mode? Check session. No.
   - Is it a command? No.
   - Regular message? Yes. Create blip.
6. Bot sends reply: "Created blip a3"
7. Reply travels back through Telegram to your phone
```

Usually under a second. Most of that is network latency.

## What I Took Away

- **Polling is easier, webhooks are cleaner.** Polling works anywhere. Webhooks scale better but need a public URL.
- **Middleware is an assembly line.** Order matters. Specific before generic.
- **Sessions enable conversations, but aren't always necessary.** Single-message commands are simpler when they work.
- **User ID is free authentication.** Telegram sends it. Use it.
- **ngrok bridges localhost and the internet.** Essential for webhook development.
